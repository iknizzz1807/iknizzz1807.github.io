JARVIS: Local-First Voice Agent on ESP32 Hardware

Beating the cloud API latency curse. A bidirectional C++ WebSocket bridge connecting constrained edge devices to a Python reasoning backend.

Tech Stack Python, C++, ESP32, WebSockets, ChromaDB, Whisper, Gemini
Role Full-Stack Embedded AI Developer

The Latency Problem of Cloud Assistants

Smart home hardware startups rely heavily on third-party voice assistants (Alexa, Google Home). However, piping raw audio to the cloud, waiting for speech-to-text (STT), then passing the text to an LLM for reasoning, then converting it back to audio (TTS), and finally streaming it back down to a micro-controller consistently results in a frustrating 3-second delay. It destroys the illusion of intelligence.

Furthermore, cloud-based assistants lack deep, persistent contextual memory across long periods of interaction. A true "smart" agent should remember a conversation from three weeks ago without having the entire transcript stuffed into its context window.

Architecting the Edge-to-Server Bridge

I tackled this problem by shifting the computational architecture. The goal was to build a fully functional, local-first voice assistant running on a $5 ESP32 microcontroller that felt as fast and intelligent as a high-end server.

  • C++ Bidirectional Audio Streaming: I programmed the ESP32 in C++ to capture raw I2S audio from a microphone and stream it continuously via a low-latency WebSocket connection, rather than relying on HTTP polling. The server continuously ingested this stream in real-time.
  • Python Reasoning Backend (ASR & LLM): The WebSocket server was built in Python. As audio packets arrived, a local instance of OpenAI's Whisper model (running on the host machine) transcribed the speech almost instantaneously. The transcribed text was fed into Gemini for high-speed reasoning.
  • The 5-Layer Persistent Vector Memory: To solve the amnesia problem, I architected a multi-tiered memory system using ChromaDB. Short-term memory (the immediate conversation) was kept in RAM. Long-term memories, facts, and user preferences were summarized by the LLM in the background and stored as vector embeddings in ChromaDB, allowing the agent to recall specific facts instantly when queried weeks later.

Implementation & Handling "Barge-ins"

The most difficult engineering hurdle was the "Barge-in" problem—when a user interrupts the AI while it is speaking. The C++ hardware code continuously monitored the microphone for amplitude spikes. If the user spoke while the ESP32's speaker was outputting audio, the C++ client instantly sent a "stop" packet over the WebSocket, and the Python server halted the TTS generation, preserving the newly interrupted context back into the LLM window.

Business Impact

By moving the STT/TTS routing to a dedicated local WebSocket server and using an intelligent, chunked transmission architecture, JARVIS achieved an end-to-end response latency of under 500ms. The persistent ChromaDB memory enabled deeply personalized interactions, demonstrating that high-end AI agents can be deployed on extremely constrained, cost-effective edge hardware.
