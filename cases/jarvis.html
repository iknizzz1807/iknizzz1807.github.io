Embedded AI System

JARVIS

A fully local, always-on voice assistant running on ESP32 hardware. Every component — from firmware to LLM orchestration — was built from scratch.

PythonC++ESP32WhisperGeminiChromaDBWebSocketMCP

The Problem

Commercial voice assistants (Alexa, Google Assistant, Siri) depend entirely on cloud processing. This introduces three fundamental problems:

  • Latency — each voice command makes a round trip to a remote data center, resulting in 2-5 second response times that break conversational flow.
  • Privacy — audio from your home is continuously streamed to and stored on third-party servers.
  • Vendor lock-in — you're constrained to each platform's closed ecosystem of skills, integrations, and supported hardware.

No off-the-shelf solution offered local ASR + TTS + LLM reasoning with persistent memory on embedded hardware. The challenge was fitting a multi-stage AI pipeline onto a microcontroller with 520KB of RAM while maintaining real-time responsiveness.

The Approach

The system was architected as a two-part pipeline: a lightweight C++ firmware layer on the ESP32 and a Python orchestration server on a local machine.

ESP32 Firmware (C++)

The firmware handles real-time audio capture via I2S, runs Silero VAD (Voice Activity Detection) directly on the microcontroller to detect speech start/end without waking the server, and streams audio chunks over WebSocket to the Python server. This offloading strategy keeps the firmware minimal while maintaining responsiveness.

Python Orchestration Server

The server receives audio streams and executes a multi-stage pipeline:

  • ASR — Whisper transcribes the audio to text with high accuracy.
  • LLM Reasoning — the transcription is sent to Gemini with full conversation context and any retrieved memories. The LLM generates both a spoken response and optional function calls.
  • TTS — the response text is converted back to audio and streamed to the ESP32 for playback.
  • Memory — ChromaDB stores conversation embeddings as persistent vector memory, allowing the assistant to recall context across sessions.

MCP Protocol Integration

Implemented the Model Context Protocol to enable dynamic tool/function calling. This allows the assistant to execute arbitrary actions (control smart devices, query APIs, run scripts) based on natural language commands, without hardcoding each capability.

The Outcome

  • Sub-2s end-to-end voice response on local hardware — no cloud round trips, no internet dependency for core operation.
  • Persistent conversational memory via ChromaDB vector storage — the assistant remembers context from previous sessions without re-prompting.
  • MCP protocol integration enabling dynamic tool/function calling — extensible without firmware changes.
  • Full offline capability — the entire ASR-to-TTS pipeline runs on local infrastructure, eliminating privacy concerns and vendor dependencies.
